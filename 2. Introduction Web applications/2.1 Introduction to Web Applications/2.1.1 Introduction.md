`Web applications` are interactive systems based on `client-server` architecture. Clients are mainly web interfaces with which user interacts and server is backbone which serves the data and work with database.
Examples Gmail, YouTube, Amazon, Google Docs etc

## Websites vs Web Applications
Websites were static pages serving content that didn't change with user interaction.  If information was to be updated, then those pages were need to be updated by the devs. It was web 1.0.

![[website_vs_webapps.jpg]]

Web applications are different. They are dynamic. Data changes with interaction and can be different for different people. They are called web 2.0. They are functional websites and can perform many functions that the traditional ones cannot do. Other differences include

- Being modular
- Running on any display size
- Running on any platform without being optimized

## Web Applications vs. Native Operating System Applications
Web applications are platform independent as compared to Native apps. They don't require space as they function on a remote server and can be run on any platform. Another advantage is that all users use the same version and we don't need to build for different people. They are updated on the web server and everyone will be on the same version. This dramatically reduces maintenance and support costs removing the need to communicate changes to all users individually.

On the other hand, native apps have some advantages. They use native libraries, that's why they are faster to load and interact with. Further they are more capable than web applications because they interact with OS and hardware and not restricted by browser capabilities. But this gap is being filled with hybrid approaches like PWA (progressive web apps) using modern  frameworks to run web applications using OS capabilities and making them more capable then web apps.

## Security Risks of Web Applications
Web applications attacks are prevalent and present a challenge for security since they are publicly accessible from all over the world with internet connection. There are many automated tools for scanning and  attacking a web application that, in wrong hand can cause issue. As web apps become more complicated and advance, so does the possibility of critical vulnerabilities being incorporated into their design.

A successful attack can cause losses and disruptions in the business. Since they are hosted on servers with other data including users or corporate data and other sensitive information. all of this can be compromised if the web app is successfully attacked. This is why it is critical to test web apps and reveal potential vulnerabilities and also patch and then test it again so that the issue is fixed and also doesn't introduce a new flaw.

One of the most current and widely used methods for testing web applications is the [OWASP Web Security Testing Guide](https://github.com/OWASP/wstg/tree/master/document/4-Web_Application_Security_Testing).

Firstly we will examine the front end by looking at HTML, CSS and JavaScript (the first trinity) to reveal any potential flaws such as leaking of `Sensitive data` or `Cross Site Scripting (XSS)`. Once all front end is tested, we will move to the web core functionality and the interaction of browser and server side to enumerate technologies being used and look for exploitable flaws. 

## Attacking Web Applications

Web applications provide a vast attack surface, and their dynamic nature means that they are constantly changing (and overlooked!). A relatively simple code change may introduce a catastrophic vulnerability or a series of vulnerabilities that can be chained together to gain access to sensitive data or remote code execution on the web server or other hosts in the environment, such as database servers.

A well-known vulnerability that is still quite prevalent in various types of web applications is SQL injection. This type of vulnerability arises from the unsafe handling of user-supplied input. It can result in access to sensitive data, reading/writing files on the database server, and even remote code execution.

We often find SQL injection vulnerabilities on web applications that use Active Directory for authentication. While we can usually not leverage this to extract passwords (since Active Directory administers them), we can often pull most or all Active Directory user email addresses, which are often the same as their usernames. This data can then be used to perform a [password spraying](https://us-cert.cisa.gov/ncas/current-activity/2019/08/08/acsc-releases-advisory-password-spraying-attacks) attack against web portals that use Active Directory for authentication such as VPN or Microsoft Outlook Web Access/Microsoft O365. A successful password spray can often result in access to sensitive data such as email or even a foothold directly into the corporate network environment.
